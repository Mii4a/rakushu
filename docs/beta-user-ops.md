# 初期βの対象入力と運営ルール

## 目的
初期βで受ける求人入力の範囲を揃え、thin-input と parser bug を混同しない。

## 前提
- いまは public beta ではなく guided private beta
- 最優先は「求人を貼れば採点できること」と「採点基準を共有できること」
- すべての求人媒体・求人文に完全対応している前提では案内しない

## 初期βで強く受けたい入力
以下は優先案内しやすい。

1. 日本語の求人本文がまとまって貼られている
2. 募集要項、勤務条件、休日、給与、福利厚生などがある程度含まれる
3. 求人媒体から本文をそのまま貼っている
4. ユーザーが比較や見極めで困っている文脈が明確
5. 事務、営業、企画、Web更新、一般的なオフィスワーク系の求人

## 初期βで弱い入力
以下は受けてもよいが、案内や期待値調整を先に入れる。

1. teaser 的な短文だけの求人
2. prose 寄りで条件がほぼ本文に出ない求人
3. 画像依存・PDF依存で本文テキストが薄い求人
4. 条件が別タブ・別ページに散っている求人
5. 英語中心、専門記号だらけ、テンプレ崩れが強い求人

## 初期βで原則対象外に近いもの
- ほぼ見出ししかない求人
- 勤務条件が外部リンク前提で本文に載っていない求人
- OCRしないと読めない入力だけ
- 1件の求人ではなく会社紹介文や採用LP全体だけ

## 判定ルール
### A. 受ける
- 本文に比較材料が十分ある
- 困りごとが比較・見極め・応募整理に近い
- guided beta の学びになりやすい

### B. 条件つきで受ける
- 本文はあるが薄い
- 一部項目しか出ていない
- `本文未記載` や `要確認` が増える可能性を先に伝える

### C. 今は深追いしない
- ほぼ teaser / prose only
- 比較材料が本文にほぼない
- parser 改善前に、まず cohort 外または運営メモ扱いにする

## 運営メモ
- `本文未記載` が多いケースは、まず parser failure ではなく source thinness を疑う
- `要確認` が多いケースは、文言揺れ・構造崩れ・抽出不安定の候補として見る
- 初期βでは「再現性のある学びが取れる入力」を優先する
- 例外的に対象外入力を受ける場合は、比較-grade sign-off では別 cohort 候補としてメモする

## ユーザー向け説明の基本文
- まだ少人数の案内制βです
- すべての求人文で完全対応ではありません
- 本文に比較材料が少ない場合、`本文未記載` や `要確認` が増えることがあります
- 困りごとの強い人から順番に案内します
