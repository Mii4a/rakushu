# βフィードバック回収ループ

## 目的
β参加者の反応を、単なる感想ではなく次の改善判断に変える。

## 回収対象
1. β登録フォームの定量
2. 自由記述の困りごと
3. 実利用時の詰まりポイント
4. `本文未記載 / 要確認` への違和感
5. 保存・比較・応募管理まで進んだか

## 週次レビューの流れ
### 1. intake確認
- 新規登録件数
- currentStatus の分布
- topProblemCategory の分布
- interviewOptIn 比率

### 2. 利用行動確認
- demo開始率
- 求人本文貼り付け率
- 解析完了率
- 保存率
- 再解析率

### 3. 定性分類
各フィードバックを以下のどれかに入れる。
- Expectation mismatch: 期待値が高すぎた / 何がβか伝わっていない
- Thin input: 元本文に比較材料が少ない
- Parser miss: 本文にあるのに拾えていない
- Rubric confusion: 採点理由や missing-item 表示の意味が伝わらない
- Workflow gap: 保存、比較、応募管理への導線が弱い
- Trust issue: 判定の信用形成が弱い

### 4. 次アクション決定
- 文言で解けるなら UI copy 修正へ
- 対象入力の問題なら β対象ルールの更新へ
- 本文にあるのに拾えていないなら parser backlog へ
- rubric説明不足なら help copy / docs へ

## 毎週見るKPI
- βフォーム送信数
- フォーム送信率
- ヒアリング協力率
- デモ開始率
- 解析完了率
- 保存率
- `本文未記載` / `要確認` に関する不満件数

## エスカレーション条件
以下のどれかに当たったらその週の優先論点に上げる。

1. expectation mismatch が 3件以上連続
2. thin-input ではないのに parser miss が 3件以上
3. `本文未記載 / 要確認` の意味が分からない反応が複数出る
4. 解析完了後に保存まで進まないケースが高止まりする
5. 同じ求人媒体で再現する崩れが続く

## 記録の持ち方
1. raw feedback はそのまま残す
2. 必ず上の定性バケットを1つ付ける
3. next action を1行で書く
4. parser issue にするなら、本文に根拠があるかを明記する

## 判断原則
- まず expectation と input scope で説明できるかを見る
- その後で parser 改善が必要かを見る
- 1件の強い声より、同じ型の反復を優先する
